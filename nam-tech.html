01 — File Structure

Five files. No build step.

The entire application ships as a flat directory. No transpilation, no bundling, no node_modules. Deployed via FTP.

No framework rationale: The app targets a single primary user in the field on mobile. Framework overhead (React hydration, bundle splitting, etc.) adds latency and complexity that doesn't pay off at this scale. Vanilla JS with async/await and CSS custom properties gets the job done in ~150KB total.

03 — Service Worker

Two cache strategies, one gray tile fallback

The service worker (nam-sw.js) implements differentiated caching based on resource type — a deliberate choice driven by the offline field use case.

📄 HTML — Stale-While-Revalidate

Serve the cached shell instantly. Revalidate in the background with event.waitUntil(networkFetch). Next load gets the fresh version. Zero-latency perceived load.

🗺 Tiles — Network-First, Cache Fallback

Always try network first to prevent stale tiles when online. On failure, serve from nam-tiles-v3 cache. LRU eviction at 500 tiles (~25MB). Opaque responses rejected explicitly.

// Firefox quirk: opaque responses (status 0) must be rejected
// or cache.put() silently stores unusable blobs
const res = await fetch(request, {
  mode: 'cors',
  credentials: 'omit'  // required for cross-origin tiles in SW context
});
if (res.ok && res.type !== 'opaque') {
  cache.put(request, res.clone());
}
Gray tile fallback: When a tile is requested offline and isn't cached, rather than letting Leaflet show a broken image, the SW synthesizes a 1×1 dark gray PNG from a base64 literal and returns it as a valid 200 image/png response. The map stays visually coherent even in uncached areas.
Cache versioning: APP_CACHE = 'nam-app-v2.0.0' and TILE_CACHE = 'nam-tiles-v3' are intentionally on separate version keys. Bumping the app version doesn't evict the tile cache — tiles are expensive to re-download in the field and are still valid regardless of app version.

API calls (iNat, eBird, WMS, parcel) explicitly pass through with no caching — dynamic data should never be served stale. The pass-through is the default fetch handler case — no respondWith means the browser handles it normally.

04 — Scan Pipeline

Parallel fetches, sequential eBird, unified normalize

The scan orchestrator runs iNaturalist and eBird concurrently via Promise.all(), but eBird's two sub-fetches (recent + notable) run sequentially to avoid race conditions on the shared PHP proxy.

// runScan() — core orchestration
const fetchPromises = [
  wantedFresh ? Promise.resolve() : forceSheetSync(),  // life list (cached 1 week)
  fetchINat(lat, lng, dist, dateStr),                   // iNat (parallel)
];
if (useEbird) fetchPromises.push(fetchEBird(lat, lng, dist)); // eBird (parallel)

const results = await Promise.all(fetchPromises);

After resolution, all observations are re-derived to attach life list cross-references. This handles the edge case where forceSheetSync() ran as part of this scan and populated fullLifeList for the first time mid-flight:

// rederive() — cross-reference life list post-fetch
const rederive = o => {
  const latin = (o.sci_name || '').toLowerCase();
  const sheetRow = fullLifeList.find(w =>
    (w.Latin2 || w.latin2 || '').toLowerCase() === latin
  );
  return {
    ...o,
    is_wanted: fullWantedList.some(w => ...latin match...),
    sheet_link: sheetRow && hasPhoto(sheetRow) ? sheetRow.Link : null
  };
};

05 — Bug & Fix

The subId notable contamination bug

A subtle data modeling bug caused every bird on an eBird checklist to be marked notable if any single bird on that checklist was notable — and simultaneously caused clicking any card to filter to the entire checklist rather than that one record.

🐛 Root cause

eBird's subId is a checklist identifier — shared by every species observation recorded on that outing. The notable detection set was keyed only on subId, so any checklist containing one notable species poisoned all others on the same checklist.

The same subId was also used as the observation's unique id field, meaning filtering by id matched the whole checklist, not one record.

// BEFORE — subId alone, not unique per species
notableIds = new Set(notableData.map(n => String(n.subId)));
is_notable: notableIds.has(String(obs.subId))  // contaminated!

// AFTER — composite key: checklist + species
notableIds = new Set(
  notableData.map(n => n.subId + '|' + (n.speciesCode || n.sciName || ''))
);
is_notable: notableIds.has(
  obs.subId + '|' + (obs.speciesCode || obs.sciName || '')
)
Cascading fix: The composite key also fixed the filter contamination bug as a side effect — since each observation's id is now 'ebird-' + subId + '|' + speciesCode, clicking a card correctly filters to exactly one record rather than the whole checklist. Two bugs, one fix.
06 — State Management

Module-scoped globals, single render function

With no framework, state is a set of module-scoped variables. The pattern that keeps this maintainable: one function, applyView(), is the only path to any UI update after a scan. Nothing renders directly — everything goes through applyView().

// Core state variables
let rawData = [];           // all observations from current scan
let fullLifeList = [];      // complete life list from Google Sheet
let fullWantedList = [];    // unseen subset — derived on sync
let activeFilter = null;    // { type: 'species'|'location'|'id', value }
let activeTaxon = 3;        // iNat taxon_id (3=birds, null=all life)
let globalToggles = { ebird: false, wanted: false };
let isInitialLoad = true;   // flyToBounds vs fitBounds on map render
applyView() — the single render path: Computes getCurrentFilteredList() from rawData + activeFilter + globalToggles, then updates markers, status bar, species heatmap, hotspot list, and detail cards in one pass. Any user action that changes visible data calls applyView() — there's no other render path.

Scroll position is preserved across filter transitions: preFilterScrollPos = window.scrollY is captured before any filter is applied, and restored exactly on clear. The gallery also captures its own scroll position for the same reason.

07 — Offline Architecture

Designed for the field, not the browser tab

The offline story is first-class, not an afterthought. The primary use case: scan at home before a hike, lose cell signal on the trail, use GPS to track position against pre-loaded sightings.

🗄 Observation Cache

Every successful scan writes the full rawData[] array to localStorage as nam_obs_cache, including the scan location, active taxon, timestamp, and the highest map zoom level reached. On next load offline, this is restored in full.

📡 GPS Without Network

navigator.geolocation is satellite-based — completely independent of cell/data. Offline Reset GPS moves the location marker and pans the map at the current zoom level without triggering a rescan or disturbing cached data.

// Offline GPS reset — no rescan, no zoom change
if (!navigator.onLine) {
  if (userMarker) {
    userMarker.setLatLng([lat, lng]);  // move dot in place
  }
  map.panTo([lat, lng], { animate: true, duration: 1.5 });
  // zoom preserved — cached tiles still valid at current level
  return;
}
Zoom cap on offline map: The cache stores maxCachedZoom — the highest zoom level the user browsed while online. When offline, applyView() caps flyToBounds at Math.min(maxCachedZoom, 16) to prevent zooming into areas where tiles haven't been fetched. The SW would serve gray placeholders otherwise — this prevents that visually jarring situation.

The online/offline transition is handled symmetrically: window.addEventListener('online') restores all network-dependent UI (scan button, geocoding, geology overlay, parcel layer). offline disables them and loads the observation cache if available.

Leaflet + WMS + GeoServer overlays

The map is built on Leaflet 1.9.4 with a layered system of base tiles and optional overlays. All overlay and tile load times are instrumented via performance.now() and reported to telemetry.

WMS GetFeatureInfo pattern: When the parcel layer is active, map clicks fire a GetFeatureInfo request to the GeoServer WMS endpoint with the current BBOX, pixel dimensions, and click coordinates translated to pixel space via map.latLngToContainerPoint(). The server returns XML which is parsed with DOMParser to extract owner name, address, county, and acreage. An AbortController with a 60-second countdown cancels stale requests when the user clicks rapidly.

The parcel layer has three distinct loading states communicated via button class mutations: parcel-tile-loading (WMS tiles loading), parcel-fetching (GetFeatureInfo in flight), parcel-active (tiles loaded, ready), and parcel-error (≥3 tile errors). Each transition has its own visual treatment in CSS.

09 — Telemetry Module

Zero-dependency, batched, beacon-safe

telemetry.js is a self-contained IIFE designed as a reusable module independent of NAM! — it can be dropped into any web app. It was built specifically to not interfere with NAM!'s existing internal dlog() debug logger.

⚡ Automatic Capture

On init(): attaches window.onerror, unhandledrejection, online/offline, visibilitychange, pagehide, and beforeunload — zero instrumentation required for baseline observability.

📦 Batched Flush

Events queue in memory and flush every 3 seconds via setTimeout. Errors bypass the batch and flush immediately. On page unload, navigator.sendBeacon() delivers remaining events — fetch() is unreliable during unload.

// sendBeacon on unload — fetch() won't complete after page close
function _flushBeacon() {
  const payload = JSON.stringify({ events: _queue.splice(0) });
  navigator.sendBeacon(
    _cfg.endpoint,
    new Blob([payload], { type: 'application/json' })
  );
}

The PHP receiver appends events to a JSONL file (telemetry-YYYY-MM.jsonl) using flock() for safe concurrent writes. Monthly rotation is filename-based — no cron required. A companion telemetry-csv.php reads the JSONL and streams a CSV download for Google Sheets import.

10 — Geocoding Migration

Mapbox → Nominatim with retry fallback

The original Mapbox geocoding integration began returning 403s — the token's allowed-domain allowlist didn't include the production domain. Rather than rotating the token, geocoding was migrated to OpenStreetMap Nominatim: no API key, no rate limits for reasonable usage, same result shape for the app's purposes.

// Two-pass geocoding: exact query, then "query + USA" fallback
async function nominatimSearch(query) {
  const res = await fetch(
    `https://nominatim.openstreetmap.org/search?q=${
      encodeURIComponent(query)
    }&format=json&limit=1`,
    { headers: { 'Accept-Language': 'en' } }
  );
  return res.json();
}

let data = await nominatimSearch(input);
if (data.length === 0) {
  data = await nominatimSearch(input + ', USA');  // disambiguation retry
}
UX on failure: If both passes return empty, the modal re-opens with the user's original query preserved in the input and an inline error message injected below it. The input is focused and selected for immediate re-edit. The "Fly There" button shows a searching state during the async call and is disabled to prevent double-submission.
11 — Design Decisions

Things worth explaining

Why Google Sheets as a database

The life list is maintained by a non-technical user. Google Sheets gives them a familiar interface for marking birds seen, adding photos via Google Photos links, and managing codes. A Google Apps Script web app publishes the sheet as JSON — no backend, no database admin, no schema migrations. The 1-week localStorage TTL means updates propagate within a week without any user action.

Why iNaturalist direct + eBird proxied

iNaturalist has a public CORS-enabled API that works fine from the browser. eBird requires an API key in the request header — headers can't be set for cross-origin requests from the browser without CORS cooperation from the server. The PHP proxy is the minimal solution: 15 lines, no state, appends the header, passes through verbatim.

Why performance.now() over Date.now() for timing

performance.now() is monotonic — it's not affected by system clock adjustments and has sub-millisecond precision. For measuring fetch durations that might be 50–5000ms, this matters. Date.now() can jump backward on clock sync. All telemetry timing uses performance.now() deltas, converted to integers via Math.round() before transmission.

Why JSONL for telemetry logs

A proper JSON array requires reading the entire file, parsing it, appending, and rewriting on every event batch — even with flock(), this is expensive and fragile under concurrent requests. JSONL (one JSON object per line) allows pure append writes. The CSV exporter does a single linear read of the file to build the output — this is fast even for large logs and requires no intermediate representation.

CSS custom properties for dual theme

The entire color system lives in :root and [data-theme="light"] custom property overrides. Theme switching is a single document.documentElement.setAttribute('data-theme', t) call — no class toggling, no JS color lookups, no style injection. The browser handles repaints. Preference is persisted to localStorage and applied on initApp() before first render to avoid flash.
